milvus——整体架构

1. 架构

Milvus 遵循数据平面和控制平面分解的原则，由四个主要层组成：访问层、协调器、工作节点、存储

1.1 访问层

访问层由一组无状态代理组成，是系统的前端层，也是用户的终端。它验证客户端请求并减少返回结果：

  • 代理本身是无状态的。它使用 Nginx、Kubernetes Ingress、NodePort 和 LVS 等负载均衡组件提供统一的服务地址。
  • 代理会对中间结果进行聚合和后处理（跨分片），然后再将最终结果返回给客户端。

1.2 协调器

协调器是 Milvus 的大脑。在任何时刻，整个集群都有一个协调器在工作，负责维护集群拓扑结构、调度所有任务类型并保证集群级一致性。

协调器的主要任务：

  • DDL/DCL/TSO 管理：处理数据定义语言 (DDL) 和数据控制语言 (DCL) 请求，如创建或删除 Collections、分区或索引，以及管理timestamp Oracle (TSO) 和时间刻度签发。
  • 流服务管理：将 Write-Ahead Log (WAL) 和 Streaming Nodes 绑定，并为 Streaming Nodes 提供服务发现功能。
  • 查询管理：管理查询节点的拓扑结构和负载平衡。
  • 历史数据管理：将压缩和建立索引等离线任务分配给数据节点，并管理数据段和数据视图的拓扑结构。

1.3 工作节点

由于存储和计算分离，工作节点是无状态的，是为了系统扩展和容灾。工作节点有三种类型：

流节点

　　基于预写日志存储（WAL）提供分片级别的一致性保证与故障恢复机制。同时，该节点还负责处理增量数据的查询请求并生成查询计划（分片内）。此外，它也承担着将增量数据转换为封存数据（segments）的关键职能。

查询节点

　　查询节点从对象存储中加载历史数据，并提供历史数据查询。

数据节点

　　数据节点负责离线处理历史数据，如压缩和建立索引。

1.4 存储

负责数据的持久化。它包括元数据存储、日志代理和对象存储。

元数据存储

　　元数据存储用于管理元数据快照，例如 collection的 schema 和消息消费位置。此类数据需保障高可用性、强一致性及事务支持。因此 Milvus 选择 etcd 作为元数据存储方案，同时基于 etcd 实现服务注册与健康检查功能。

对象存储

　　对象存储负责保存日志快照文件、标量与向量数据的索引文件以及中间查询结果。Milvus 默认采用 MinIO 作为对象存储，同时支持无缝对接 S3 存储。

WAL 存储

　　先写日志（WAL）存储是分布式系统中数据持久性和一致性的基础。在提交任何更改之前，首先要将其记录在日志中，以确保在发生故障时，可以准确恢复到之前的位置。

　　常见的 WAL 实现包括 Kafka、Pulsar 和 Woodpecker。与传统的基于磁盘的解决方案不同，Woodpecker 采用云原生的零磁盘设计，直接写入对象存储。

2. 数据操作

2.1 数据插入

　　在 Milvus 中，您可以选择一个集合（collection）使用的分片（shard）数量——每个分片会映射到一个虚拟通道（vchannel）。Milvus 会将每个虚拟通道分配到物理通道（pchannel），而每个物理通道会绑定到特定的流处理节点（Streaming Node）。

　　代理在数据验证之后，会根据指定的分片路由规则，将写入的消息拆分成各个分片的数据包。

　　然后将一个分片（vchannel）的写入数据发送到 pchannel 对应的流节点。

　　流节点为每个数据包分配 Timestamp Oracle(TSO），以此建立操作的全局顺序关系。在将有效载荷写入 WAL 前，系统会先执行一致性检查。一旦数据被持久化提交至预写日志，即可保证零丢失——即使发生系统崩溃，流式处理节点也能通过重放预写日志完整恢复所有待处理操作。

　　与此同时，流式处理节点还会异步地将已提交的 WAL 条目切分为独立段（segment）。段分为两种类型：

  • Growing segment：尚未持久化到对象存储的数据；
  • Sealed segment：所有数据都已持久化到对象存储中，密封段的数据是不可变的。

　　从增长数据段过渡到密封数据段的过程称为刷新。一旦流节点摄取并写入了该分段的所有可用 WAL 条目，流节点就会触发刷新。

2.2 建立索引

索引建立由数据节点执行。为了避免频繁为数据更新建立索引，Milvus 将 Collections 进一步划分为多个分段，每个分段都有自己的索引。

　　Milvus 支持为每个向量字段、标量字段和主键字段构建索引。索引构建的输入和输出都与对象存储有关：数据节点将要建立索引的日志快照从段落（在对象存储中）加载到内存，反序列化相应的数据和元数据以建立索引，索引建立完成后序列化索引，并将其写回对象存储。

　　索引构建主要涉及向量和矩阵操作，因此是计算和内存密集型操作。向量因其高维特性，无法用传统的树形索引高效地建立索引，需采用更适合该领域的聚类或图结构等更成熟的技术。无论其类型如何，建立索引都涉及大规模向量的大量迭代计算，如 K均值聚类或图遍历。

　　此外，Milvus 还支持标量过滤与主键查询。为了提高查询效率，系统还内置布隆过滤器索引、哈希索引、树状索引、倒排索引等多种索引，并计划引入位图索引、粗糙索引等外部索引类型。

2.3 数据查询

　　数据查询指的是在指定的 Collections 中搜索与目标向量最接近的 k 个向量，或搜索与向量在指定距离范围内的所有向量。向量会连同其相应的主键和字段一起返回。

　　Milvus 中的 Collections 分成多个分段；流节点加载 growing segments 并维护实时数据，而查询节点则加载封存的分段。

　　当收到 query/search 请求时，proxy 会将请求广播给负责相关分片的所有流节点，以进行并发搜索。

　　每个流节点生成一个查询计划，搜索其本地 growing data，同时调用查询节点检索历史结果，并聚合为分片级别的结果。

　　最后，代理收集所有分片结果，将其合并为最终结果，并返回给客户端。

　　当流节点上的增长分段被刷新为密封分段时，或者当数据节点完成压缩时，协调器会启动移交操作，将增长数据转换为历史数据。然后，协调器会在所有查询节点上平均分配密封分段，平衡内存使用率、CPU 开销和分段数，并释放冗余分段。

３. 安装

支持三种部署方式：

  • Milvus Lite：是一个 Python 库，可嵌入应用程序。
  • Milvus Standalone：单机版。
  • Milvus Distributed：分布式版本。

分布式版本部署方式：

３.1 helm 方式

添加 Milvus Helm 资源库

helm repo add zilliztech https://zilliztech.github.io/milvus-helm/
# 查看已添加的资源库
helm repo list
# 查看所有可用版本
helm search repo zilliztech/milvus --versions
1. 在线安装
helm install my-milvus zilliztech/milvus \
  --set image.all.tag=v2.6.0 \
  --set pulsarv3.enabled=false \
  --set woodpecker.enabled=true \
  --set streaming.enabled=true \
  --set indexNode.enabled=false

等同于：

helm pull zilliztech/milvus --version=5.0.0

helm install my-milvus ./milvus-5.0.0.tgz \
  --set image.all.tag=v2.6.0 \
  --set pulsarv3.enabled=false \
  --set woodpecker.enabled=true \
  --set streaming.enabled=true \
  --set indexNode.enabled=false

此命令的作用：

  • 使用Woodpecker作为消息队列（建议使用，以减少维护工作）
  • 启用新的流节点组件以提高性能
  • 禁用传统的索引节点（其功能现在由数据节点处理）
  • 禁用 Pulsar，改用 Woodpecker

Milvus 2.6.x 中的架构变更：

  • 消息队列：现在推荐使用Woodpecker（与 Pulsar 相比，减少了基础设施维护工作）
  • 新组件：引入流节点并默认启用
  • 合并组件：索引节点和数据节点合并为一个数据节点

更新Milvus 集群配置：

helm upgrade my-milvus zilliztech/milvus --namespace my-namespace -f values.yaml
2. 离线安装

获取 Milvus 清单：

helm template my-release zilliztech/milvus > milvus_manifest.yaml

自定义配置：

helm template -f values.yaml my-release zilliztech/milvus > milvus_manifest.yaml

下载镜像：

wget https://raw.githubusercontent.com/milvus-io/milvus/master/deployments/offline/requirements.txt
wget https://raw.githubusercontent.com/milvus-io/milvus/master/deployments/offline/save_image.py
pip3 install -r requirements.txt
python3 save_image.py --manifest milvus_manifest.yaml

加载镜像：

for image in $(find . -type f -name "*.tar.gz") ; do gunzip -c $image | docker load; done

部署 Milvus:

kubectl apply -f milvus_manifest.yaml
3. 升级运行中的 Milvus 群集
helm repo update
helm upgrade my-release zilliztech/milvus --reset-then-reuse-values

３.2 Milvus Operator

  1. 使用 Helm 安装 Milvus Operator

    helm install milvus-operator \
  -n milvus-operator --create-namespace \
  --wait --wait-for-jobs \
  https://github.com/zilliztech/milvus-operator/releases/download/v1.3.0/milvus-operator-1.3.0.tgz

  2. 部署 Milvus 群集

    kubectl apply -f https://raw.githubusercontent.com/zilliztech/milvus-operator/main/config/samples/milvus_cluster_woodpecker.yaml

  3. 查看集群状态

    kubectl get milvus my-release -o yaml

  4. 更新配置

    kubectl patch milvus my-release --type='merge'\
  -p '{"spec":{"components":{"disableMetric":false}}}'
