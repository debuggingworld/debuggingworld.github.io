LangChain、LangGraph 总结

一、基础概念篇

Q1: LangChain 和 LangGraph 的定位区别是什么？什么时候用哪个？

LangChain 是 LLM 应用开发框架，提供可组合组件（Model、Prompt、Retriever、Tool、OutputParser 等），适合线性 Chain 拼接场景。

LangGraph 是 LLM Agent 的编排框架和运行时，专注有状态的、长时间运行的 Agent 工作流，提供图结构的状态机、持久化、人工介入等能力。

维度 LangChain LangGraph
核心抽象 Chain（线性管道） StateGraph（有向图 + 状态机）
状态管理 无内置状态 内置 State + Reducer + Checkpointer
循环/分支 不原生支持 原生条件边、循环
人工介入 无原生支持 interrupt / breakpoint / 人工审批
持久化 无 Checkpointer（Sqlite/Postgres/Redis）
适合场景 RAG 管道、简单 Chain 多步推理 Agent、审批流、复杂工作流
依赖关系 独立 可独立使用，不依赖 LangChain

选型原则：线性流水线用 LangChain；需要循环、状态、条件分支、持久化 → LangGraph。企业级 Agent 必选 LangGraph。

Q2: LangGraph 的核心抽象有哪些？各自职责？

6 个核心抽象：

  1. State（状态） — 图的共享数据结构，用 TypedDict 或 Pydantic BaseModel 定义。所有节点读写同一份状态。
  2. Node（节点） — 接收 State，返回 State 更新的函数。def my_node(state: State) -> dict
  3. Edge（边） — 定义节点间的执行顺序。普通边（确定性）和条件边（动态路由）。
  4. StateGraph（状态图） — 将节点和边组装成有向图的容器，compile() 后生成可执行图。
  5. Checkpointer（检查点） — 持久化每一步的状态快照，支持断点恢复和时间旅行调试。
  6. Reducer（归约器） — 定义状态字段的合并策略（覆盖 vs 追加）。默认覆盖，messages 字段用 add_messages 追加。
from typing import Annotated
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages

class AgentState(TypedDict):
    messages: Annotated[list, add_messages]  # Reducer: 追加而非覆盖
    next_step: str                            # 默认 Reducer: 覆盖

Q3: State 的 Reducer 机制是什么？为什么需要它？

问题：多节点对同一 State 字段更新时，如何合并？默认是覆盖（新值替换旧值），但有些字段需要追加（如对话消息列表）。

Reducer 就是合并策略函数，通过 Annotated[type, reducer_fn] 声明。

from operator import add
from typing import Annotated

class State(TypedDict):
    messages: Annotated[list, add_messages]  # 消息追加
    artifacts: Annotated[list, add]          # 列表拼接
    score: float                              # 默认覆盖

内置 Reducer： - add_messages — 智能追加消息（支持同 id 消息的更新/删除） - add（operator.add）— 列表拼接

自定义 Reducer： 

def merge_metadata(existing: dict, new: dict) -> dict:
    result = {**existing, **new}  # 浅合并
    return result

class State(TypedDict):
    metadata: Annotated[dict, merge_metadata]

面试要点：Reducer 解决了分布式图计算中的状态合并问题，灵感来自 Pregel 的顶点状态聚合。

Q4: 条件边（Conditional Edge）如何实现？和普通边的区别？

普通边：确定性路由，A 执行完一定到 B。 

graph.add_edge("node_a", "node_b")

条件边：根据 State 动态选择下一个节点，这是实现 Agent 循环和分支的关键。 

def route_by_intent(state: AgentState) -> str:
    last_msg = state["messages"][-1]
    if last_msg.tool_calls:
        return "tool_node"
    return "__end__"

graph.add_conditional_edges(
    "agent",           # 源节点
    route_by_intent,   # 路由函数，返回节点名
    {"tool_node": "tool_node", "__end__": END}  # 可选的映射表
)

关键区别： - 条件边函数只接收 State，必须返回目标节点名的字符串 - 映射表是可选的，不提供时函数返回值直接作为节点名 - 条件边是实现 ReAct 循环（agent → tool → agent → ...）的基础

Q5: LangGraph 的执行模型是什么？和传统 DAG 有什么不同？

灵感来源：Pregel（Google 图计算框架）+ Apache Beam。

传统 DAG（如 Airflow）：无环有向图，节点执行一次，单向流动。

LangGraph 的图： - 允许环（循环）— 这是 Agent "思考-行动-观察" 循环的基础 - Pregel 超步模型 — 每一步所有节点并行计算，然后同步状态 - 状态驱动 — 节点间不直接传递消息，而是通过共享 State 间接通信 - 持久化每一步 — 每个超步结束后 Checkpoint 保存完整状态

执行流程： 

1. 初始化 State
2. 执行当前步骤的所有就绪节点（可并行）
3. 节点返回状态更新 → 通过 Reducer 合并
4. Checkpoint 保存
5. 根据边定义确定下一步就绪节点
6. 重复 2-5 直到到达 END

二、架构设计篇

Q5.1: Checkpointer 是必须的吗？节点间怎么触发？

不是必须的。Checkpointer 只在需要跨 invoke 恢复时才需要。

节点间自动流转是运行时行为，跟持久化无关——一个节点 return 后，运行时根据边定义自动调度下一个节点，这是内存中的同步过程，不需要任何持久化。

# 无 Checkpointer — 一次 invoke 从 START 跑到 END
graph = workflow.compile()
result = graph.invoke({"messages": [...]})  # 节点间自动触发，一口气跑完

跨 invoke 恢复才需要持久化——两次 invoke 之间进程可能已退出，必须靠 Checkpointer 把 State 存下来。

# 有 Checkpointer — 支持暂停恢复
graph = workflow.compile(checkpointer=checkpointer)
result = graph.invoke(input, {"configurable": {"thread_id": "t1"}})  # 执行到 interrupt 停下
# ... 几小时后 ...
result = graph.invoke(None, {"configurable": {"thread_id": "t1"}})  # 从 Checkpoint 恢复继续

选型判断：

场景 是否需要 Checkpointer 原因
单次请求内跑完 不需要 节点间运行时自动调度
Human-in-the-Loop 需要 暂停后进程退出，恢复时需读取状态
多轮对话 需要 跨 invoke 保持上下文
长时间运行任务 需要 服务可能重启，需断点恢复
时间旅行调试 需要 回退到历史状态重放

企业审批流典型场景：用户发起请假 → 审批人几小时后才点通过 → 服务进程早已处理其他请求 → 必须持久化。

面试要点：区分"节点间自动流转"（运行时调度，无需持久化）和"跨请求恢复"（需要 Checkpointer）。面试官常拿这个考察你对 LangGraph 运行模型的理解深度。

Q6: LangGraph 如何实现 Human-in-the-Loop？企业级场景怎么用？

三种机制，由浅入深：

1. Breakpoint（断点） — 在指定节点前暂停，等待人工审批 

from langgraph.checkpoint.memory import MemorySaver

checkpointer = MemorySaver()
graph = workflow.compile(
    checkpointer=checkpointer,
    interrupt_before=["approval_node"]  # 在此节点前暂停
)

# 执行到断点后恢复
result = graph.invoke(
    None,  # 不传新输入
    config={"configurable": {"thread_id": "thread-1"}},
)

2. interrupt() 函数 — 在节点内部动态触发暂停 

from langgraph.types import interrupt

def review_node(state: State):
    decision = interrupt("需要人工审核此文档")  # 暂停，返回值在恢复时提供
    return {"approved": decision}

3. 人工修改 State — 暂停后可修改状态再继续 

# 获取当前状态快照
snapshot = graph.get_state(config)
# 人工修改后更新
graph.update_state(config, {"approved": True}, as_node="review_node")
# 继续执行
graph.invoke(None, config)

企业场景：审批流程（请假、报销）→ Agent 整理后 interrupt → 人工在飞书卡片点击确认 → resume 执行。

Q7: Checkpointer 机制详解，生产环境怎么选？

作用：持久化每一步的完整状态快照，实现断点恢复和时间旅行。

核心能力： - 自动保存每个超步后的 State - 支持从任意历史步骤恢复（时间旅行） - 多线程隔离（thread_id） - 配合 interrupt 实现人工介入

实现选择：

Checkpointer 适用场景 特点
MemorySaver 开发调试 内存存储，进程退出即丢
SqliteSaver 单机部署 轻量持久化
AsyncSqliteSaver 单机异步 异步版 Sqlite
PostgresSaver 生产首选 高可用、支持并发
AsyncPostgresSaver 生产首选 异步版 Postgres
RedisSaver 高性能场景 低延迟

生产建议： - 必须用持久化 Checkpointer（Postgres），不能用 MemorySaver - thread_id 作为业务会话标识（如飞书消息的 chat_id） - 定期清理历史 Checkpoint，避免存储膨胀

from langgraph.checkpoint.postgres import PostgresSaver

with PostgresSaver.from_conn_string("postgresql://...") as checkpointer:
    graph = workflow.compile(checkpointer=checkpointer)
    result = graph.invoke(input, config={"configurable": {"thread_id": "user-123"}})

Q8: LangGraph 的内存体系是什么？短期记忆 vs 长期记忆？

短期记忆（Short-term Memory）： - 就是 Checkpoint 中的 State，随 thread_id 隔离 - 单次会话的上下文，会话结束可清理 - 通过 graph.get_state() 获取

长期记忆（Long-term Memory）： - 跨会话、跨线程持久化 - 使用 LangGraph Store（BaseStore） - 支持命名空间（namespace）组织记忆 - 适合用户偏好、历史摘要、知识积累

from langgraph.store.memory import InMemoryStore
from langgraph.store.base import BaseStore

store = InMemoryStore()  # 生产用 PostgresStore

# 在节点中使用
def memory_node(state: State, *, store: BaseStore):
    # 存储记忆
    store.put(("user", "preferences"), "theme", {"value": "dark"})
    # 检索记忆
    pref = store.get(("user", "preferences"), "theme")
    return {"messages": [...]}

面试要点： - 短期记忆 = 会话内 State（自动管理） - 长期记忆 = 跨会话 Store（需主动读写） - 企业 Agent 需两者结合：会话内保持上下文，跨会话积累用户画像

Q9: 子图（Subgraph）怎么用？什么时候需要？

子图 = 嵌套在父图中的独立 StateGraph，用于模块化和复用。

核心问题：父子图 State 可能不同，如何传递？

方式一：共享 Key（简单场景） 

# 父 State 和子 State 有相同 key 的字段，自动映射
class ParentState(TypedDict):
    messages: Annotated[list, add_messages]
    query: str

class ChildState(TypedDict):
    messages: Annotated[list, add_messages]  # 共享 key，自动映射
    result: str

方式二：显式转换函数（复杂场景） 

# 父 → 子
def parent_to_child(parent: ParentState) -> ChildState:
    return {"messages": parent["messages"], "query": parent["query"]}

# 子 → 父
def child_to_parent(child: ChildState, parent: ParentState) -> ParentState:
    return {"messages": child["messages"]}

使用场景： - RAG 子图（检索 + 重排 + 过滤）被多个 Agent 复用 - 审批子图（提交 → 审批 → 通知）作为公共模块 - 团队协作：不同团队独立开发子图，再组装

Q10: 并行执行和 Map-Reduce 怎么做？

并行执行：多个节点无依赖时自动并行（Pregel 超步模型）。 

# A 完成后，B 和 C 并行执行
graph.add_edge("A", "B")
graph.add_edge("A", "C")
graph.add_edge("B", "D")  # D 等待 B、C 都完成
graph.add_edge("C", "D")

Map-Reduce：动态生成并行节点处理列表数据。 

def map_step(state: State):
    # 为每个 item 创建一个 Send 指令
    return [Send("process_node", {"item": item}) for item in state["items"]]

graph.add_conditional_edges("fan_out", map_step)
# process_node 的所有结果通过 Reducer 汇聚到 State

注意：Send 允许向节点发送自定义 State，是实现 Map-Reduce 的关键。

三、Agent 模式篇

Q11: 用 LangGraph 实现 ReAct Agent 的标准模式？

ReAct（Reason + Act）是最经典的 Agent 模式：LLM 决策 → 调用工具 → 观察结果 → 继续推理。

from langgraph.graph import StateGraph, MessagesState, START, END
from langgraph.prebuilt import ToolNode, tools_condition

# 1. 定义工具
tools = [search_tool, calculator_tool]
tool_node = ToolNode(tools)

# 2. 定义 Agent 节点
def agent_node(state: MessagesState):
    response = llm.bind_tools(tools).invoke(state["messages"])
    return {"messages": [response]}

# 3. 构建图
graph = StateGraph(MessagesState)
graph.add_node("agent", agent_node)
graph.add_node("tools", tool_node)
graph.add_edge(START, "agent")
graph.add_conditional_edges("agent", tools_condition)  # 有 tool_calls → tools，否则 → END
graph.add_edge("tools", "agent")  # 工具结果返回 Agent 继续推理

app = graph.compile()

tools_condition 是内置条件边函数：检测最后一条消息是否有 tool_calls，有则路由到工具节点，无则结束。

Q12: 预构建的 create_react_agent 和手动构建的区别？

from langgraph.prebuilt import create_react_agent

app = create_react_agent(llm, tools, prompt="你是一个企业助手")

一行代码完成 Q11 的全部工作。内部就是 StateGraph + agent_node + ToolNode + tools_condition。

何时手写： - 需要自定义节点（审批、日志、降级） - 需要条件分支（不同意图走不同路径） - 需要人工介入（interrupt） - 需要子图编排

何时用预构建： - 简单问答/工具调用场景 - 快速原型验证 - 不需要自定义流程控制

Q13: 多 Agent 协作模式有哪些？LangGraph 怎么实现？

三种主流模式：

1. Supervisor（监督者模式） 一个中央 Agent 分发任务给专业子 Agent。 

Supervisor → [ResearchAgent, CodeAgent, WriteAgent]

2. Swarm（群智模式） Agent 之间可互相转交控制权，无中心节点。 

Agent A → handoff → Agent B → handoff → Agent C

3. Hierarchical（层级模式） 多级 Supervisor，上层分配，下层执行。

LangGraph 实现要点： 

# Supervisor 模式
def supervisor(state: State) -> str:
    # LLM 决定路由到哪个子 Agent
    return route_to_agent

graph.add_conditional_edges("supervisor", supervisor, {
    "research": "research_agent",
    "code": "code_agent",
    "finish": END
})

# 子 Agent 完成后回到 Supervisor
graph.add_edge("research_agent", "supervisor")
graph.add_edge("code_agent", "supervisor")

Q14: LangGraph 的 Functional API 是什么？和 StateGraph 的区别？

Functional API（v0.2+）提供更 Pythonic 的声明式写法，用装饰器代替显式图构建。

from langgraph.func import entrypoint, task

@task
def research(query: str) -> str:
    return search_tool.invoke(query)

@task
def summarize(text: str) -> str:
    return llm.invoke(f"总结：{text}")

@entrypoint(checkpointer=MemorySaver())
def agent(query: str):
    result = research(query)
    summary = summarize(result)
    return summary
维度 StateGraph Functional API
风格 显式图定义（add_node/add_edge） 装饰器声明
灵活度 高（条件边、并行、子图） 中（适合线性/简单分支）
可视化 天然支持（LangSmith） 支持
适用 复杂状态机 简单工作流

面试建议：说清两者区别，强调企业级复杂场景用 StateGraph。

四、工程实践篇

方案三：容器化部署

CMD ["langgraph", "up", "--port", "8000"]

生产架构： 

飞书/前端 → API Gateway → LangGraph Server → LLM API
                            ↕
                        Postgres (Checkpoint + Store)
                            ↕
                        MCP Server (工具调用)

Q16: MCP（Model Context Protocol）在 LangGraph Agent 中怎么集成？

MCP 是 Anthropic 提出的工具调用标准协议，让 Agent 能发现和调用外部工具。

集成方式：

from langchain_mcp import MCPToolkit

# 1. 连接 MCP Server
async with MCPToolkit(url="http://mcp-server:3000/sse") as toolkit:
    tools = toolkit.get_tools()

    # 2. 将 MCP 工具注册到 Agent
    app = create_react_agent(llm, tools)

    # 3. 执行
    result = await app.ainvoke({"messages": [("user", "查询我的审批状态")]})

企业场景：通过 MCP Server 暴露内部系统接口（OA 审批、CRM 查询、知识库检索），Agent 自动发现并调用。

MCP Server 开发： 

from mcp.server import Server
from mcp.types import Tool

server = Server("enterprise-tools")

@server.tool()
async def query_approval(approval_id: str) -> dict:
    """查询审批单状态"""
    return await oa_client.get_approval(approval_id)

@server.tool()
async def search_knowledgebase(query: str) -> list:
    """搜索企业知识库"""
    return await kb_client.search(query)

Q17: Agent 的流式输出怎么做？为什么重要？

为什么重要：LLM 响应慢（数秒），流式输出让用户实时看到部分结果，体验提升显著。

三种流式模式：

# 1. 消息流（逐 token 输出）
for event in graph.stream(input, config, stream_mode="messages"):
    print(event.content, end="", flush=True)

# 2. 更新流（每个节点完成时输出）
for event in graph.stream(input, config, stream_mode="updates"):
    print(f"节点 {event['node']} 完成: {event['output']}")

# 3. 值流（每步输出完整 State）
for event in graph.stream(input, config, stream_mode="values"):
    print(event)

飞书场景：stream_mode="messages" → 逐 token 更新卡片消息，实现打字机效果。

Q18: 如何设计 Agent 的错误处理和降级策略？

1. 节点级重试 

from tenacity import retry, stop_after_attempt

@retry(stop=stop_after_attempt(3))
def call_llm(state: State):
    return llm.invoke(state["messages"])

2. 降级路由 

def safe_agent(state: State):
    try:
        return primary_llm.invoke(state["messages"])
    except Exception:
        return fallback_llm.invoke(state["messages"])

3. 条件边降级 

def route_with_fallback(state: State) -> str:
    if state.get("error_count", 0) >= 3:
        return "human_escalation"  # 升级到人工
    if state.get("llm_failed"):
        return "fallback_node"
    return "normal_node"

4. Checkpoint 恢复 

# 从最近成功状态恢复
last_state = graph.get_state(config)
graph.update_state(config, last_state.values, as_node="last_success_node")
graph.invoke(None, config)

五、性能与调优篇

Q19: LangGraph 的性能瓶颈在哪？如何优化？

瓶颈： 1. LLM 调用延迟（最大瓶颈） 2. 状态序列化/反序列化（大 State 时明显） 3. Checkpoint 写入（高频场景下 I/O 密集）

优化策略：

策略 方法
减少 LLM 调用 缓存相似查询结果、合并小工具调用
精简 State 只存必要字段，大文档用引用 ID
异步执行 使用 async 节点 + AsyncCheckpointer
并行化 无依赖节点并行执行（Pregel 超步）
流式输出 stream_mode="messages" 减少等待感
消息裁剪 trim_messages 控制上下文窗口

消息裁剪： 

from langgraph.graph.message import trim_messages

def agent_node(state: State):
    trimmed = trim_messages(state["messages"], max_tokens=4000, strategy="last")
    return {"messages": [llm.invoke(trimmed)]}

Q20: 如何控制 Agent 的 Token 消耗？

  1. 消息窗口裁剪 — 保留最近 N 轮对话，超出部分摘要
  2. System Prompt 精简 — 避免冗长指令
  3. 工具描述精简 — 只传必要工具，描述简洁
  4. 结构化输出 — 用 with_structured_output 减少冗余 token
  5. 缓存 — 相同输入缓存 LLM 响应（LangSmith 缓存或 Redis）
from langchain_core.messages import trim_messages

trimmed = trim_messages(
    messages,
    max_tokens=4000,
    strategy="last",        # 保留最新的
    token_counter=llm,      # 用 LLM 的 tokenizer 计数
    start_on="human",       # 从 human 消息开始裁剪
    include_system=True,    # 始终保留 system 消息
)

七、企业级状态机 Demo

场景：企业审批助手（请假审批流）

流程： 

用户发起请假 → Agent 补全信息 → 查询余额 → 生成审批单 → 人工审批 → 通知结果
                    ↓                                    ↓
               信息不足，追问                          驳回，可修改重提

# ============ State 定义 ============
from typing import Annotated, Literal
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langgraph.checkpoint.postgres import PostgresSaver
from langgraph.types import interrupt

class ApprovalState(TypedDict):
    messages: Annotated[list, add_messages]
    leave_type: str | None           # 请假类型
    start_date: str | None           # 开始日期
    end_date: str | None             # 结束日期
    reason: str | None               # 请事由
    remaining_days: float | None     # 剩余额度
    approval_id: str | None          # 审批单号
    approval_status: str | None      # approved / rejected / pending
    retry_count: int                 # 重试次数


# ============ 节点函数 ============

def info_extract_node(state: ApprovalState) -> dict:
    """从用户消息中提取结构化信息，缺失则追问"""
    extracted = llm.with_structured_output(LeaveInfo).invoke(
        f"从以下消息提取请假信息：{state['messages'][-1].content}"
    )
    missing = []
    if not extracted.leave_type: missing.append("请假类型")
    if not extracted.start_date: missing.append("开始日期")
    if not extracted.end_date: missing.append("结束日期")
    if not extracted.reason: missing.append("请假事由")

    if missing:
        return {"messages": [AIMessage(content=f"请补充以下信息：{', '.join(missing)}")]}

    return {
        "leave_type": extracted.leave_type,
        "start_date": extracted.start_date,
        "end_date": extracted.end_date,
        "reason": extracted.reason,
    }


def balance_check_node(state: ApprovalState) -> dict:
    """查询请假余额"""
    balance = oa_client.get_leave_balance(
        user_id=state["messages"][0].additional_kwargs["user_id"],
        leave_type=state["leave_type"],
    )
    days_requested = calc_days(state["start_date"], state["end_date"])
    if days_requested > balance:
        return {
            "messages": [AIMessage(
                content=f"额度不足：申请 {days_requested} 天，剩余 {balance} 天"
            )],
            "remaining_days": balance,
        }
    return {"remaining_days": balance}


def create_approval_node(state: ApprovalState) -> dict:
    """创建审批单并暂停等待审批"""
    approval_id = oa_client.create_approval(
        type=state["leave_type"],
        start=state["start_date"],
        end=state["end_date"],
        reason=state["reason"],
    )
    # 发送飞书审批卡片给审批人
    feishu.send_approval_card(approval_id, state)

    # 暂停等待人工审批
    decision = interrupt(f"审批单 {approval_id} 等待审批")
    return {
        "approval_id": approval_id,
        "approval_status": decision,  # "approved" or "rejected"
    }


def notify_node(state: ApprovalState) -> dict:
    """通知审批结果"""
    if state["approval_status"] == "approved":
        msg = f"审批通过！单号 {state['approval_id']}"
    else:
        msg = f"审批被驳回，可修改后重新提交。单号 {state['approval_id']}"
    feishu.send_message(state["messages"][0].additional_kwargs["chat_id"], msg)
    return {"messages": [AIMessage(content=msg)]}


# ============ 路由函数 ============

def route_after_extract(state: ApprovalState) -> str:
    """信息提取后：信息完整 → 查余额，缺失 → 追问后结束"""
    if state["leave_type"] and state["start_date"] and state["end_date"]:
        return "balance_check"
    return END  # 追问后等用户下次输入

def route_after_balance(state: ApprovalState) -> str:
    """余额查询后：余额够 → 创建审批，不够 → 结束"""
    days_requested = calc_days(state["start_date"], state["end_date"])
    if days_requested > (state["remaining_days"] or 0):
        return END
    return "create_approval"

def route_after_approval(state: ApprovalState) -> str:
    """审批后：通过 → 通知，驳回 → 通知（可重提）"""
    return "notify"


# ============ 构建图 ============

graph = StateGraph(ApprovalState)

graph.add_node("info_extract", info_extract_node)
graph.add_node("balance_check", balance_check_node)
graph.add_node("create_approval", create_approval_node)
graph.add_node("notify", notify_node)

graph.add_edge(START, "info_extract")
graph.add_conditional_edges("info_extract", route_after_extract)
graph.add_conditional_edges("balance_check", route_after_balance)
graph.add_edge("create_approval", "notify")
graph.add_edge("notify", END)

# 编译：启用 Checkpoint + 审批前断点
checkpointer = PostgresSaver.from_conn_string(DATABASE_URL)
app = graph.compile(
    checkpointer=checkpointer,
    interrupt_before=["create_approval"],  # 可选：在创建审批前暂停确认
)


# ============ 飞书 Webhook 接入 ============

@app.route("/feishu/webhook", methods=["POST"])
def handle_feishu_message():
    event = request.json
    chat_id = event["message"]["chat_id"]
    user_msg = event["message"]["content"]

    config = {"configurable": {"thread_id": chat_id}}
    result = app.invoke(
        {"messages": [HumanMessage(content=user_msg, additional_kwargs={"chat_id": chat_id, "user_id": ...})]},
        config,
    )
    return {"status": "ok"}

@app.route("/feishu/approval_callback", methods=["POST"])
def handle_approval_callback():
    """飞书审批卡片回调"""
    action = request.json
    thread_id = action["chat_id"]
    decision = action["value"]["decision"]  # approved / rejected

    config = {"configurable": {"thread_id": thread_id}}
    # 恢复被 interrupt 暂停的图
    app.update_state(config, {"approval_status": decision}, as_node="create_approval")
    app.invoke(None, config)
    return {"status": "ok"}

核心设计要点：

  1. State 是核心 — 所有业务数据在 State 中流转，节点间不直接通信
  2. 条件边驱动分支 — 信息完整度、余额、审批结果决定路由
  3. interrupt + update_state 实现人工审批 — 暂停 → 飞书卡片 → 回调恢复
  4. Checkpoint 保证可靠性 — 每步持久化，服务重启不丢状态
  5. thread_id 隔离会话 — 飞书 chat_id 作为 thread_id，多轮对话天然隔离

八、高频追问速答

问题 核心答案
LangGraph 能不用 LangChain 吗？ 能，LangGraph 可独立使用，LLM 可用任意提供商
StateGraph vs MessageGraph？ StateGraph 自定义 State；MessageGraph 是 StateGraph(MessagesState) 的语法糖
如何调试 Agent？ LangSmith 可视化追踪 + get_state() 查看状态快照
如何限流？ 节点内加 rate limiter；或在 Gateway 层统一限流
Agent 如何做评测？ LangSmith Datasets + Evaluator；编写 gold-answer 用例，对比输出
多租户怎么做？ thread_id 隔离；Store 用 namespace 按租户分区；Checkpointer 按租户分库
和 CrewAI 区别？ CrewAI 是高层框架，角色驱动；LangGraph 是底层编排，图驱动。企业级选 LangGraph，可控性更强
和 Dify 区别？ Dify 是低代码平台，拖拽构建；LangGraph 是代码框架，灵活度更高。复杂逻辑选 LangGraph
LangGraph Platform 是什么？ LangGraph 的生产部署方案，提供 Server + API（Assistants/Threads/Crons），支持持久化和流式
如何做 Agent 版本管理？ graph.compile() 产出的 CompiledGraph 可序列化；配合 LangSmith 追踪版本
