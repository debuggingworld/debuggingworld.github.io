记录面试高频问题的深度回答，定位：大厂面试，重在"讲清楚"而非"背得多"。

Q1: 什么是 MCP？用 Python 写一个服务端和客户端的 Demo

MCP（Model Context Protocol）是 Anthropic 在 2024 年底发布的一个开放协议，解决的核心问题是：LLM 如何以标准化方式与外部系统交互。

为什么需要 MCP

在 MCP 之前，每接入一个外部系统（数据库、API、文件系统），就要写一套自定义的 tool 描述 + 调用逻辑。不同 Agent 框架（LangChain、CrewAI、Dify）各自实现工具协议，互不兼容。MCP 做的事类似 HTTP 之于网络通信——定义一套统一的消息格式和交互模式，让任何 LLM 客户端都能调用任何 MCP Server 暴露的能力，无需关心实现细节。

MCP 的能力模型

能力 类比 说明
Resources GET 接口 服务端暴露的只读数据，LLM 按需读取（如配置文件、数据库记录）
Tools POST 接口 LLM 可调用的函数，有副作用（如发邮件、查订单、执行 SQL）
Prompts 模板 可复用的 prompt 模板，支持参数化
Sampling 回调 服务端请求客户端 LLM 做推理——反转调用方向。工具执行到一半需要 LLM 判断时（如"这段 SQL 结果是否需要进一步分析"），不用重新起一轮对话，Server 直接向 Client 发起 sampling 请求
Roots 工作目录 客户端告知服务端可访问的根目录/资源边界，Server 据此决定暴露哪些资源。用于限制 Server 的访问范围

传输层

  • stdio：客户端以子进程方式启动服务端，通过 stdin/stdout 通信。仅适合本地开发调试，生产环境不用。
  • Streamable HTTP：MCP 协议 2025-03-26 版本引入，取代旧的 HTTP+SSE 传输。服务端提供单一 HTTP 端点，客户端通过 POST 发送 JSON-RPC 请求，服务端可返回 JSON 响应或开启 SSE 流式推送。支持独立部署、多客户端复用、会话管理、断线重连，是生产环境首选。
  • SSE（已废弃）：旧版 HTTP+SSE 传输（2024-11-05 协议版本），已被 Streamable HTTP 取代。新项目不应使用。

实际选型：生产环境用 Streamable HTTP。stdio 只在本地调试时用。

选型依据： - stdio 的优势是零网络延迟、零部署依赖、进程级隔离天然安全。适合 CLI 工具链场景（如 Claude Desktop 本地调 MCP Server、IDE 插件内嵌调用） - Streamable HTTP 的优势是可独立部署扩缩容、多客户端复用同一 Server、支持 OAuth 2.1 鉴权、支持断线重连。适合服务化场景（企业内部 MCP Server、SaaS 平台对外暴露工具能力） - 选型的核心判断是调用场景而非简单的一句"生产用 HTTP"——如果 Agent 和 MCP Server 始终同机运行，stdio 反而更简单可靠

交互流程

Client (LLM 应用)                    Server (MCP Server)
     │                                      │
     │  ── initialize ──────────────────►   │  (握手，交换能力)
     │  ◄── initialize response ─────────   │
     │                                      │
     │  ── tools/list ──────────────────►   │  (发现可用工具)
     │  ◄── [tool1, tool2, ...] ────────   │
     │                                      │
     │  ── tools/call {name, args} ────►    │  (调用工具)
     │  ◄── {content, isError} ─────────   │
     │                                      │
     │  ── resources/list ──────────────►   │  (发现可用资源)
     │  ◄── [resource1, ...] ───────────   │
     │                                      │
     │  ── resources/read {uri} ───────►    │  (读取资源)
     │  ◄── {contents} ──────────────────   │

所有通信基于 JSON-RPC 2.0。

服务端 Demo（一个进程，多个 Streamable HTTP 端点）

# server.py — 一个 HTTP 进程，按业务域挂多个 Streamable HTTP 端点
from mcp.server.fastmcp import FastMCP
from starlette.applications import Starlette
from starlette.routing import Mount
import uvicorn

# ── 订单服务 ──
order_mcp = FastMCP("order-service")


@order_mcp.tool()
def query_order(order_id: str) -> dict:
    """查询订单信息。

    Args:
        order_id: 订单编号，格式为 ORD-xxxxx

    Returns:
        订单详情，包含状态、金额、创建时间
    """
    orders = {
        "ORD-10001": {"status": "已发货", "amount": 299.0, "created": "2026-05-18"},
        "ORD-10002": {"status": "待付款", "amount": 1580.0, "created": "2026-05-20"},
    }
    if order_id in orders:
        return orders[order_id]
    return {"error": f"订单 {order_id} 不存在"}


@order_mcp.tool()
def cancel_order(order_id: str, reason: str) -> dict:
    """取消订单。

    Args:
        order_id: 订单编号
        reason: 取消原因

    Returns:
        操作结果
    """
    return {"order_id": order_id, "action": "cancelled", "reason": reason}


# ── 通知服务 ──
notify_mcp = FastMCP("notify-service")


@notify_mcp.tool()
def send_message(user_id: str, text: str) -> dict:
    """发送飞书文本消息给指定用户。

    Args:
        user_id: 用户飞书 ID，格式 ou_xxxxx
        text: 消息内容

    Returns:
        发送结果
    """
    return {"user_id": user_id, "status": "sent"}


@notify_mcp.tool()
def send_card(user_id: str, title: str, content: str) -> dict:
    """发送飞书卡片消息给指定用户。

    Args:
        user_id: 用户飞书 ID，格式 ou_xxxxx
        title: 卡片标题
        content: 卡片正文

    Returns:
        发送结果
    """
    return {"user_id": user_id, "status": "sent", "type": "card"}


# ── 一个 Starlette 应用，挂载多个 Streamable HTTP 端点 ──
app = Starlette(
    routes=[
        Mount("/mcp/order", app=order_mcp.streamable_http_app()),
        Mount("/mcp/notify", app=notify_mcp.streamable_http_app()),
    ],
)

if __name__ == "__main__":
    uvicorn.run(app, host="0.0.0.0", port=9001)

启动后两个业务域的 Streamable HTTP 端点： - 订单服务：http://localhost:9001/mcp/order/mcp - 通知服务：http://localhost:9001/mcp/notify/mcp

Agent 按需连接不同端点，只拿到对应业务域的工具，不会一次性拉取全部。

客户端 Demo（Streamable HTTP 模式）

# client.py — 通过 Streamable HTTP 连接远程 MCP Server
import asyncio
from mcp import ClientSession
from mcp.client.streamable_http import streamable_http_client


async def main():
    # 连接远程 MCP Server（Streamable HTTP 模式）
    async with streamable_http_client("http://localhost:9001/mcp/order/mcp") as (read_stream, write_stream, _):
        async with ClientSession(read_stream, write_stream) as session:
            # 1. 初始化
            await session.initialize()

            # 2. 发现工具
            tools_result = await session.list_tools()
            print("可用工具:")
            for tool in tools_result.tools:
                print(f"  - {tool.name}: {tool.description}")

            # 3. 调用工具
            result = await session.call_tool("query_order", {"order_id": "ORD-10001"})
            print(f"\n查询结果: {result.content}")

            # 4. 取消订单
            result = await session.call_tool(
                "cancel_order",
                {"order_id": "ORD-10002", "reason": "用户主动取消"},
            )
            print(f"取消结果: {result.content}")

            # 5. 读取资源
            resources = await session.list_resources()
            print(f"\n可用资源: {[r.uri for r in resources.resources]}")

            config = await session.read_resource("config://max_retry")
            print(f"配置 max_retry = {config.contents}")


if __name__ == "__main__":
    asyncio.run(main())

Tool Description

Tool Description 就是 @mcp.tool() 函数的 docstring，以 Demo 中 query_order 为例：

完整链路：docstring → FastMCP 自动提取填入 tool.description 字段 → 客户端 list_tools() 通过 MCP 协议返回 → LLM 看到工具名 + 描述 + 参数 Schema → 据此决定"调不调"和"参数怎么填"。

mcp 包说明

mcp — Anthropic 官方 MCP Python SDK：

pip install mcp

核心模块：

模块 用途
mcp.server.fastmcp.FastMCP 高级服务端 API，装饰器风格，自动处理 docstring 提取、类型注解转 JSON Schema、传输层启动
mcp.server.Server 低级服务端 API，适合需要自定义 middleware 或流式 tool result 的场景
mcp.client.streamable_http.streamable_http_client Streamable HTTP 传输层客户端，生产环境首选
mcp.client.sse.sse_client SSE 传输层客户端（已废弃，向后兼容保留）
mcp.client.stdio.stdio_client stdio 传输层客户端，仅用于本地调试
mcp.ClientSession 客户端会话管理，封装 initialize / list_tools / call_tool 等方法

FastMCP 是 mcp 包内置的高级封装，不需要额外安装。

MCP 和 Google ADK 整合

Google ADK 2.0 核心概念

ADK（Agent Development Kit）是 Google 开源的 Python Agent 框架，核心抽象：

概念 说明
Agent 最小执行单元，绑定 model + instruction + tools
Workflow 图编排引擎，支持条件路由、扇出/扇入、循环、重试、Human-in-the-Loop
Task Agent Agent 间委托机制，支持多轮任务模式和单轮受控输出
Tool Agent 可调用的函数，支持原生 FunctionTool 和 MCP 工具集

ADK 集成 MCP 的方式

ADK Agent — 通过 MCPToolset 接入 MCP 工具（Streamable HTTP 模式）

# approval_agent.py
from google.adk import Agent
from google.adk.tools.mcp_toolset import MCPToolset, StreamableHTTPConnectionParams


# 定义如何连接远程 MCP Server（Streamable HTTP 模式）
oa_mcp_params = StreamableHTTPConnectionParams(
    url="http://oa-service.internal:9001/mcp/oa/mcp",
)

# 创建 Agent，自动加载 MCP Server 中的所有工具
root_agent = Agent(
    name="approval_assistant",
    model="gemini-2.5-flash",
    instruction="""你是一个企业审批助手。你的职责：

1. 帮员工查询假期余额
2. 帮员工提交请假申请
3. 查询审批进度

工作流程：
- 用户提出请假需求时，先查询假期余额确认是否足够
- 余额足够时提交申请，余额不足时告知用户并建议其他假期类型
- 用户询问进度时查询审批状态

注意：只有用户明确要求请假时才提交申请，不要主动提交。""",
    tools=[MCPToolset(connection_params=oa_mcp_params)],
)

面试关键区分点

  1. Tool Description 是 MCP 开发中最被低估的技能。LLM 根据工具描述决定是否调用和如何传参，描述不清晰 = Agent 瞎调用。写法要点：动词开头、参数约束明确、给出格式示例。
  2. MCP 与 Function Calling 的关系：Function Calling 是 LLM 的能力（模型层面），MCP 是工具的协议（工程层面）。MCP 管的是"工具怎么注册、怎么发现、怎么调用"，不管"模型怎么决定调用哪个"。
  3. 企业级关注点：鉴权（MCP 规范已正式包含 OAuth 2.1 授权框架，基于 PKCE + 动态客户端注册 + 授权服务器元数据发现）、沙箱（工具执行的隔离）、审计（调用日志与追踪）。这些是面试中区分"做过 Demo"和"做过生产"的关键。
  4. 企业级 MCP Server 的组织方式：不是每个业务域单独部署一个服务，而是一个服务承载多组工具
    • 方式一：一个 FastMCP 实例 + 工具名前缀（oa_query_leave、feishu_send_message），最简单
    • 方式二：一个 HTTP 进程挂多个 Streamable HTTP 端点（/mcp/oa、/mcp/feishu），逻辑隔离但物理统一部署
    • 方式三：配置驱动动态注册，不同调用方看到不同工具集
    • 逻辑上按业务域组织工具，物理上可以是一个部署单元
  5. 生产环境的核心问题不是"能不能跑通"，而是：
    • 鉴权：MCP Server 怎么验证调用者身份？→ 在 MCP 协议层加 token 传递，Server 端校验
    • 限流：避免 Agent 疯狂调用工具刷爆下游系统 → 在 MCP Server 端加 rate limiting
    • 可观测：每次工具调用的入参、出参、耗时、是否出错 → 加 before_tool_call / after_tool_call 钩子打日志或上报到 LangSmith / LangFuse
    • 降级：MCP Server 挂了怎么办 → ADK Agent 的 instruction 中要写明兜底话术，避免无限重试

Q2: 在实际 MCP 服务端/客户端开发中遇到了哪些问题？

按影响程度排序，从"最容易踩坑"到"最难解决"。

问题1：工具列表过大导致上下文爆炸

现象：一个企业级 MCP Server 可能有 30-50 个工具，每个工具的 name + description + inputSchema 平均 200 token，50 个工具就是 10000 token——每次请求 LLM 都要带这 10000 token，即使用户只是问了一句"你好"。

更严重的后果：工具越多，LLM 选择准确率越低。实测工具数超过 20 个时，选择错误率明显上升；超过 40 个时，LLM 经常"困惑"——要么选错工具，要么该调用时不调用。

解法（从简单到彻底）：

层级 做法 效果
部署层 一个进程多个 Streamable HTTP 端点，Agent 按业务域连接 每个 Agent 只看到 5-10 个工具
协议层 用底层 Server API 实现 list_tools 动态过滤 根据调用方身份/角色返回不同工具子集
Agent 层 两步调用：先调 search_tools(query) 选工具，再调具体工具 工具从"全量展示"变成"按需检索"
框架层 在 Agent instruction 中限定可用工具范围 由 LLM 自我约束，但不够可靠

最推荐的是部署层 + 协议层组合：Streamable HTTP 端点做粗粒度隔离，list_tools 动态过滤做细粒度控制。这既减少了 token 消耗，又提高了 LLM 选择准确率。

协议层动态过滤的实现：list_tools 不是静态快照，而是一次请求——服务端完全可以根据请求上下文动态返回不同的工具列表。关键在于 HTTP 请求时通过 Authorization Header 携带身份信息（如 Bearer <token>），服务端解析后在 list_tools 时按身份过滤。

FastMCP 的高级 API 不直接暴露 request 上下文给 list_tools，需要用底层 mcp.server.Server API：

from mcp.server import Server

server = Server("enterprise-tools")

# 所有工具的完整定义
ALL_TOOLS = [oa_query_leave_tool, feishu_send_tool, admin_reset_tool, ...]

@server.list_tools()
async def list_tools(ctx) -> list[Tool]:
    """每次调用都动态决定返回哪些工具，不是启动时写死的"""
    # 从连接上下文中拿到调用方身份
    role = ctx.session.client_info.metadata.get("role", "user")
    tenant = ctx.session.client_info.metadata.get("tenant", "default")

    # 按身份过滤
    tools = [t for t in ALL_TOOLS if t.name.startswith(f"{tenant}_")]
    if role != "admin":
        tools = [t for t in tools if not t.name.startswith("admin_")]
    return tools

@server.call_tool()
async def call_tool(name: str, arguments: dict) -> list:
    # 先校验：这个身份是否有权调这个工具
    # 防止客户端绕过 list_tools 直接构造 call_tool 请求
    ...

这样不同调用方连接同一个 MCP Server，看到的工具列表完全不同： - 普通用户连接 → 只看到 OA 查询 + 飞书发消息 - 管理员连接 → 额外看到 admin_ 开头的系统管理工具 - A 租户连接 → 只看到 A 租户的业务工具

这是企业级 SaaS 化 MCP Server 的核心能力——工具列表不是"有什么就暴露什么"，而是"你有权看什么才返回什么"。

问题2：鉴权——如何保护 MCP Server 不被未授权调用

现象：MCP 规范已定义了基于 OAuth 2.1 的标准授权框架，但具体实现仍需服务端配置。如果 MCP Server 未启用授权，任何人只要知道 URL 就能调用你的工具。这在企业内网勉强能忍，公网部署绝对不行。

解法（从标准到过渡）：

  1. 遵循 MCP 规范的 OAuth 2.1 授权流程（推荐，新项目首选）：Server 未授权返回 HTTP 401 → 客户端走 OAuth 2.1 + PKCE 获取 access token → 后续请求 Authorization Header 携带 Bearer token。支持动态客户端注册（RFC7591）和授权服务器元数据发现（RFC8414）。

  2. 在 HTTP middleware 层拦截（过渡方案，尚未完全对接 OAuth 流程时使用）：在 MCP 端点前加 middleware 校验 token，校验通过后把身份信息注入 request state 供 list_tools 使用。关键实现：token 校验在 HTTP 层完成，MCP 协议层只负责根据身份过滤工具列表。

关键点：MCP 规范要求 Bearer token 必须包含在每次 HTTP 请求的 Authorization Header 中（即使同属一个逻辑会话），服务端通过 Mcp-Session-Id 跟踪会话状态。

问题3：工具执行结果过大，反向污染上下文

现象：query_order 返回一个订单还好，但如果是 search_orders 返回 200 条订单记录呢？每条 500 字，结果就是 100000 字直接灌入 LLM 上下文，可能直接撑爆 token 限制。

这是 MCP 开发中最容易被忽视的问题——大家都在关注"工具描述占多少 token"，却忽略了"工具返回值占多少 token"。

解法：

  1. Server 端截断：在工具实现中限制返回条数和字段 

    @mcp.tool()
def search_orders(status: str, limit: int = 5) -> dict:
    """搜索订单。默认返回最多5条，如需更多请使用 page 参数翻页。"""
    results = db.query(status=status)
    return {"total": len(results), "results": results[:limit], "has_more": len(results) > limit}

  2. Agent 端摘要：在 after_tool_call 钩子中对大结果做 LLM 摘要，只把摘要送入后续上下文

  3. 分页 + 追问模式：工具返回摘要 + "是否需要查看详情"，让 LLM 引导用户翻页，而不是一次性返回所有数据

问题4：LLM 选错工具或传错参数

现象：Agent 有 query_order 和 cancel_order 两个工具，用户说"帮我查一下订单"，LLM 却调了 cancel_order。或者调了 query_order 但把 order_id 传成了用户名。

根因：不是代码 bug，是工具描述和参数描述不够精确。LLM 完全靠 description 理解工具，描述有歧义就会出错。

解法：不是靠代码，而是靠"写好 description"这项被严重低估的技能。

# 差：LLM 可能误判
@mcp.tool()
def cancel_order(order_id: str) -> dict:
    """取消订单"""
    ...

# 好：场景限定 + 参数约束 + 副作用警告 + 格式示例
@mcp.tool()
def cancel_order(order_id: str, reason: str) -> dict:
    """取消指定订单。此操作不可逆，仅在用户明确要求取消订单时使用。
    不要在用户只想查询订单时误调此工具。

    Args:
        order_id: 订单编号，格式 ORD-xxxxx，如 ORD-10001
        reason: 取消原因，必填，至少5个字

    Returns:
        取消结果；若订单不存在或已取消则返回错误信息
    """
    ...

四个要素： - 场景限定："仅在用户明确要求取消时使用"——防止误调 - 副作用警告："此操作不可逆"——让 LLM 在不确定时倾向不调用 - 参数约束：格式、示例、必填——防止传错参 - 边界情况：订单不存在/已取消——让 LLM 知道会有什么异常

更深一层：LLM 作为 API 消费者的根本性挑战

选错工具只是表象，根因是：你的 API 消费者不是确定性程序，而是概率模型。传统 API 的消费者是人写的代码，调用行为完全确定；MCP 工具的消费者是 LLM，每次调用都是概率采样。具体表现：

  1. 幻觉调用：调用不存在的工具名、编造参数值（如 order_id 传成"用户想查的订单"这种自然语言）
  2. 不一致性：相同输入，不同次调用可能选不同工具。工具越多越不稳定
  3. 过度调用：用户只问了一句话，LLM 连续调了 5 个工具"以防万一"

应对策略（分层防御）：

层级 策略 作用
Description 层 精确的工具描述（上文四个要素） 减少误判源头
校验层 Server 端严格校验参数类型和格式，拒绝非法值 兜底，防止幻觉参数穿透到业务系统
确认层 高风险操作（删除、退款）Agent 先向用户确认再执行 防止不可逆误操作
评测层 自动化评测工具选择准确率（给定 query + 工具列表，验证 LLM 是否选对） 持续监控，description 改动后回归

面试话术：这不是 MCP 的问题，也不是某个模型的问题，而是"LLM 作为 API 消费者"这个范式带来的根本性挑战——你写的每一行工具代码，都要假设调用者是一个会犯错的概率模型，而不是一个确定性的程序。

问题5：MCP Server 的生命周期管理

现象：MCP Server 进程挂了，Agent 端怎么感知？自动重连吗？正在执行的 call_tool 怎么办？

现状：MCP 协议没有定义标准的健康检查和重连机制。Streamable HTTP 连接断开后，客户端需要自己实现重连逻辑（规范定义了基于 Last-Event-ID 的断线重连机制，但客户端需主动实现）。

解法：

# 客户端侧：带重连和重试的 Streamable HTTP 连接
import asyncio
from mcp.client.streamable_http import streamable_http_client
from mcp import ClientSession

async def with_retry(url: str, callback, max_retries: int = 3):
    """带重试的连接。业务逻辑通过 callback 传入，确保 session 在 async with 内使用。"""
    for attempt in range(max_retries):
        try:
            async with streamable_http_client(url) as (read, write, _):
                async with ClientSession(read, write) as session:
                    await session.initialize()
                    return await callback(session)
        except Exception as e:
            wait = 2 ** attempt  # 指数退避
            print(f"连接失败，{wait}s 后重试: {e}")
            await asyncio.sleep(wait)
    raise ConnectionError(f"无法连接 {url}")

Server 端侧：用 Kubernetes 的 liveness/readiness probe + 进程守护（supervisor/systemd），保证 Server 挂了能自动拉起。

问题6：工具调用缺乏可观测性

现象：Agent 调了一个工具，返回了结果，但你在日志里什么都看不到。工具被调了几次？哪个工具最慢？哪个工具经常返回错误？一概不知。

解法：在 MCP Server 端加中间层做日志和指标采集。

import time
import logging
from functools import wraps

logger = logging.getLogger("mcp.audit")

def audit_tool(func):
    """装饰器：记录工具调用的入参、出参、耗时、是否出错"""
    @wraps(func)
    async def wrapper(*args, **kwargs):
        tool_name = func.__name__
        start = time.time()
        try:
            result = await func(*args, **kwargs)
            duration = time.time() - start
            logger.info(f"tool={tool_name} args={kwargs} duration={duration:.3f}s status=ok")
            # 上报指标到监控系统（Prometheus / OpenTelemetry）
            TOOL_LATENCY.labels(tool=tool_name).observe(duration)
            TOOL_CALLS.labels(tool=tool_name, status="ok").inc()
            return result
        except Exception as e:
            duration = time.time() - start
            logger.error(f"tool={tool_name} args={kwargs} duration={duration:.3f}s error={e}")
            TOOL_CALLS.labels(tool=tool_name, status="error").inc()
            raise
    return wrapper

# 使用：给所有需要审计的工具加装饰器
@mcp.tool()
@audit_tool
async def query_order(order_id: str) -> dict:
    ...

生产环境必须关注的核心指标： - 工具调用次数（按工具名、按状态 ok/error 分维度） - 工具调用延迟（P50/P95/P99） - 工具返回大小（发现哪个工具返回值过大） - LLM 工具选择准确率（需要人工标注或自动评测）

问题7：工具版本升级时的兼容性

现象：query_order 要加一个必选参数 region，改完之后所有接入的 Agent 都报错——因为它们的 instruction 和历史对话里还是旧参数格式。

解法：

  • 参数只加不删：新参数用 Optional + 默认值，不破坏旧调用 

    @mcp.tool()
def query_order(order_id: str, region: str = "cn") -> dict:
    # region 有默认值，旧调用方式不受影响
    ...

  • 工具名带版本（最后手段）：query_order_v2，旧工具保留一段时间后下线

  • 在 description 中标注版本和变更：让 LLM 知道参数变化

这不是 MCP 独有的问题，是所有 API 版本管理的通病。但 MCP 的特殊性在于：你的 API 消费者不是人写的代码，而是 LLM——它不会看 changelog，只看 description。所以 description 的及时更新比传统 API 文档更重要。

总结：面试中怎么呈现

不要流水账式地罗列问题，而是按"问题 → 影响 → 解法 → 权衡"的结构来讲。面试官想听到的不是"我遇到了 XX 问题"，而是"我遇到了 XX 问题，它的影响是 YY，我尝试了 A/B/C 三种解法，最终选了 B，因为 ZZ"。

最容易被追问的三个问题，建议重点准备： 1. 上下文爆炸（问得最多）——解法的层次感（部署层 / 协议层 / Agent 层） 2. 鉴权（企业场景必问）——MCP 规范已包含 OAuth 2.1，实际落地怎么做 3. 工具选择准确率（最体现深度）——根因是 description 质量，不是模型能力

Q3: A2A 协议是什么？和 MCP 是什么关系？

A2A（Agent-to-Agent Protocol）是 Google 在 2025 年 4 月发布的开放协议，解决的核心问题是：不同框架/平台构建的 Agent 之间如何互操作。

为什么需要 A2A

企业里不可能一个 Agent 干所有事。审批 Agent、知识库 Agent、数据分析 Agent 可能是不同团队用不同框架（LangGraph / CrewAI / ADK / Dify）构建的。没有 A2A 之前，Agent 之间互调只能硬编码 API 对接，每对接一次就写一套。A2A 做的事和 MCP 类似——定义统一的交互协议，让任意两个 Agent 能发现彼此、理解彼此的能力、协商任务执行方式。

A2A 的核心概念

概念 说明
Agent Card Agent 的"名片"，JSON 格式，描述能力、认证方式、支持的交互模式。放在 .well-known/agent.json，类似 robots.txt
Task A2A 交互的核心单元，有状态机（submitted → working → completed/failed/canceled/rejected/input-required/auth-required）
Message Task 中的消息，分 role=user 和 role=agent
Part 消息内容片段，支持 TextPart / FilePart / DataPart（结构化数据）
Artifact Task 产生的产出物（报告、分析结果等），一个 Task 可产出多个 Artifact
Push Notification 长任务完成后主动通知 Client Agent，不需要轮询

Task 状态机

submitted → working → completed
                  ├── failed
                  ├── canceled
                  ├── rejected
                  ├── input-required (需要 Client 补充信息，Human-in-the-loop)
                  └── auth-required (需要 Client 提供认证)

input-required 是关键状态——Remote Agent 执行到一半发现缺信息，可以暂停等 Client Agent（或背后的人类）补充后继续。比 MCP 的"一次性调用、一次性返回"更灵活。

交互流程（JSON-RPC 绑定）

Client Agent                          Remote Agent
     │                                      │
     │  1. GET /.well-known/agent.json ──►  │  (读 Agent Card，了解能力)
     │  ◄── {name, capabilities, ...} ───   │
     │                                      │
     │  2. POST /  (JSON-RPC: tasks/send) ► │  (发起 Task)
     │  ◄── {taskId, status: "working"} ──  │
     │                                      │
     │  3. POST /  (JSON-RPC: tasks/get) ─► │  (查询 Task 状态)
     │  ◄── {taskId, status: "completed",   │
     │        artifacts: [...]} ──────────   │
     │                                      │
     │  长任务场景：                          │
     │  2. POST /  (JSON-RPC: tasks/send,   │
     │     带 pushConfig)                   │
     │  ◄── {taskId, status: "working"}      │
     │  ... (Remote Agent 干活中)             │
     │  ◄── Push Notification ──────────     │  (主动通知完成)
     │  3. POST /  (JSON-RPC: tasks/get) ─► │  (拿结果)
     │  ◄── {artifacts: [...]} ──────────    │

说明：A2A 默认绑定是 JSON-RPC 2.0 over HTTP(S)，所有方法（tasks/send、tasks/get 等）通过向同一端点发送 JSON-RPC 请求调用，方法名在 body 中指定。也支持 REST 和 gRPC 绑定。

A2A 与 MCP 的关系——互补，不是竞争

维度 MCP A2A
交互对象 Agent ↔︎ 工具/外部系统 Agent ↔︎ Agent
交互模式 请求-响应（支持服务端通知推送） Task 状态机（支持异步长任务）
发现机制 list_tools Agent Card（.well-known/agent.json）
输出 工具返回值 Task + Artifact（可多次产出）
协作模式 单 Agent 调多个工具 多 Agent 对等协作
通信协议 JSON-RPC over stdio/Streamable HTTP JSON-RPC 2.0 over HTTP(S)（默认绑定，也支持 REST/gRPC）

完整架构中的位置：

用户 → Agent A ──MCP──► OA 系统（查审批）
              │
              └──A2A──► Agent B ──MCP──► 数据平台（跑 SQL）
                              │
                              └──A2A──► Agent C ──MCP──► 飞书（发通知）
  • Agent 通过 MCP 接入业务系统的工具
  • Agent 通过 A2A 和其他 Agent 协作
  • 两者解决不同层面的问题，共同构成企业级 Agent 架构

具体实现

Google 官方提供 A2A Python SDK：

from a2a.server import A2AServer, AgentCard, Skill

card = AgentCard(
    name="data-analysis-agent",
    description="数据分析 Agent，支持 SQL 查询和报表生成",
    skills=[
        Skill(id="sql_query", name="SQL 查询", description="执行 SQL 并返回结果"),
        Skill(id="generate_report", name="报表生成", description="根据查询结果生成分析报表"),
    ],
)

server = A2AServer(agent_card=card)

@server.on_task
async def handle_task(task):
    ...
# Client Agent — 发现 Remote Agent，发起 Task

# 1. 读取 Agent Card

# 2. 发起 Task

# 3. 获取结果
if task.status.state == "completed":
    for artifact in task.artifacts:
        print(artifact.parts)

面试要点

  1. A2A 和 MCP 是互补关系，不是竞争。面试官如果问"A2A 会不会取代 MCP"，答案是"不会——它们解决不同层面的问题，MCP 管 Agent↔︎工具，A2A 管 Agent↔︎Agent"
  2. A2A 的 Task 状态机比 MCP 的请求-响应更灵活，特别是 input-required 状态支持 Human-in-the-loop，适合需要人工确认的企业流程
  3. Agent Card 是 A2A 的发现机制，类似微服务的服务注册。未来可能出现企业内部的 Agent Registry，各部门注册自己的 Agent，其他部门直接发现和调用
  4. A2A 目前生态成熟度远不如 MCP，落地案例还很少。面试中重点是讲清楚"解决什么问题、和 MCP 的关系"，而不是假装深度用过

Q4: Claude Code / OpenClaw 这些工具里面多个 Agent 之间是如何调用的？

回答

这两个工具的 Agent 间调用都不是用 A2A，而是各自有内部机制。A2A 是跨平台 Agent 互操作的协议，同工具内的 Agent 协作不需要走标准协议——同进程内直接调更快。

Claude Code：Agent Tool 模式

Claude Code 本身就是一个 Agent，它通过内置的 Agent Tool 生成子 Agent：

主 Agent (Claude Code)
  │
  ├── Agent Tool → 生成 Explore 子 Agent（只读搜索）
  ├── Agent Tool → 生成 Plan 子 Agent（只规划不执行）
  ├── Agent Tool → 生成 general-purpose 子 Agent（通用）
  ...

工作方式： - 主 Agent 决定：分析任务后，通过调用 Agent Tool 生成一个子 Agent - 上下文传递：通过 prompt 参数把任务描述传给子 Agent - 子 Agent 独立运行：有自己的工具集（如 Explore 类型只有只读工具）、独立的上下文窗口 - 结果回传：子 Agent 完成后，返回一条消息给主 Agent，主 Agent 继续工作

本质是函数调用——和调 grep、调 Read 没有本质区别，只不过"工具"变成了另一个 Agent。同进程内，不走网络协议。

OpenClaw：Session Spawn + Multi-Agent Routing

OpenClaw 有两套机制，解决不同场景：

场景1：同一 Gateway 内的 Agent 委托 — Session Spawn

主 Agent (审批助手)
  │
  ├── sessions_spawn(runtime: "subagent") → 子 Agent (数据分析)
  │     └── 返回结果给主 Agent
  │
  └── sessions_spawn(runtime: "subagent") → 子 Agent (通知)
        └── 返回结果给主 Agent

工作方式： 1. 主 Agent 通过 sessions_spawn 工具 spawn 一个子 Agent（runtime: "subagent" 在沙箱内运行） 2. 子 Agent 运行在自己的 session 中，但共享同一个 Gateway 3. 父子 Agent 之间通过 session tools（sessions_send、sessions_history）通信 4. 子 Agent 的输出流式回传给父 Agent 5. 子 Agent 可以访问父 Agent 授权的工具

场景2：多 Agent 隔离共存 — Multi-Agent Routing

Gateway (一个进程)
  │
  ├── Agent "工作助手" ← Binding: WhatsApp(工作号)
  │     └── 独立 workspace / session / auth
  │
  ├── Agent "生活助手" ← Binding: WhatsApp(个人号)
  │     └── 独立 workspace / session / auth
  │
  └── Agent "数据分析" ← Binding: Discord(bot1)
        └── 独立 workspace / session / auth

这不是"Agent 之间互相调用"，而是多个独立 Agent 共存于一个 Gateway，各自服务不同的渠道/用户，互不干扰。Agent 间通信需显式启用 tools.agentToAgent 并配置白名单。

场景3：外部 Harness 接入 — ACP（Agent Client Protocol）

OpenClaw Gateway
  │
  ├── ACP 连接 → Claude Code CLI（作为外部 Harness）
  ├── ACP 连接 → Gemini CLI（作为外部 Harness）
  └── ACP 连接 → Cursor（作为外部 Harness）

ACP（Agent Client Protocol）是 OpenClaw 的协议，用于让外部编码工具作为"运行时"接入。核心用法： - openclaw acp：IDE/客户端通过 stdio 与 OpenClaw 通信，OpenClaw 将请求转发到 Gateway session - /acp spawn <harness>：OpenClaw 启动外部 Harness（Claude Code、Codex、Gemini CLI 等）作为 ACP 会话 - ACPX：@openclaw/acpx 插件，管理 Harness 启动、模型控制、MCP 桥接，将 OpenClaw 的工具暴露给 ACP Harness

注意区分：OpenClaw 的 ACP（Agent Client Protocol）与 IBM BeeAI 的 ACP（Agent Communication Protocol，已归档合并至 A2A）是完全不同的协议，只是缩写相同。

三种方式的对比

方式 Claude Code Agent Tool OpenClaw Session Spawn A2A
通信方式 同进程函数调用 Gateway 内 session tools HTTP REST
适用场景 主 Agent 拆分子任务 主 Agent 委托子 Agent 跨平台 Agent 互操作
上下文共享 通过 prompt 传参 通过 sessions_send 流式传递 通过 Task Message
是否需要网络 不需要 不需要 需要
协议复杂度 最低 中等 最高

面试要点

  1. 同工具内的 Agent 协作用内部机制，不走 A2A——因为同进程内直接调比走 HTTP 快得多，而且不需要服务发现、鉴权等开销
  2. A2A 的价值在跨平台、跨团队场景——用 LangGraph 构建的 Agent 和用 CrewAI 构建的 Agent 需要互操作时，才需要标准协议
  3. 类比：公司内部沟通用企微/飞书（内部协议），跨公司沟通用邮件（SMTP 标准协议）。Claude Code 和 OpenClaw 的 Agent 间调用是"公司内部沟通"，A2A 是"跨公司沟通"
  4. 面试中容易被追问："那你们企业内部 Agent 协作用什么？"——如果所有 Agent 都是同一框架构建的，用框架内部机制（如 ADK 的 Task Agent、OpenClaw 的 Session Spawn）就够；如果涉及跨框架、跨团队，才引入 A2A
  5. 注意 ACP 缩写歧义：OpenClaw 的 ACP 是 Agent Client Protocol（外部 Harness 接入），IBM BeeAI 的 ACP 是 Agent Communication Protocol（Agent 间通信，已合并至 A2A），两者完全不同。面试中如果提到 ACP，要主动澄清是哪一个

Q5: MCP / ACP / A2A / ANP 四大协议的区别

各协议的核心机制

MCP：上下文注入

核心是按需注入——不用把所有信息一次性塞进 prompt，只给 AI 最关键的内容。支持文件、数据库、API 实时响应通过标准接口直接喂给 AI。企业可安全接入 SAP、Salesforce 等内部系统，读写结构化数据而不暴露底层 API。

ACP：基于 REST 的 Agent 通信协议

ACP 通过 HTTP REST API 实现 Agent 间通信，核心交互：发现（GET /agents）→ 执行（POST /runs）→ 获取结果。最有特色的设计是 Await 机制——Agent 执行到一半需要外部输入时可以暂停，等客户端补充信息后恢复执行（Human-in-the-loop）。支持同步、流式、异步三种执行模式，以及 Session 跨多次交互保持状态。

ACP 和 A2A 在通信方式上相似（都是 HTTP），区别主要在协议风格（ACP 更简洁 REST，A2A 更规范 JSON-RPC）和发现机制（ACP 用 /agents 端点，A2A 用 .well-known/agent.json）。

现状：ACP 仓库已于 2025-08-27 正式归档，官方声明"ACP is now part of A2A under the Linux Foundation"。ACP 与 A2A 概念高度重叠（Agent Manifest ≈ Agent Card，Run ≈ Task，Await ≈ input-required），社区选择 A2A 作为 Agent 间通信的主线协议。这不是代码级的合并，而是概念收敛。

注意：OpenClaw 也有一个缩写为 ACP 的协议（Agent Client Protocol），用于外部编码 Harness 接入，与 IBM BeeAI 的 ACP（Agent Communication Protocol）完全不同。面试中提到 ACP 时要主动澄清。

A2A：Agent Card + Task 状态机

每个智能体有一张 Agent Card（JSON 格式的"技能名片"），别的 AI 读卡片就知道你能干嘛、怎么找你。基于 JSON-RPC 2.0 over HTTP(S)（默认绑定），也支持 REST 和 gRPC 绑定，web-native，支持 OAuth 2.0。智能体既可以是客户端（发起任务），也可以是服务端（接任务），甚至两者兼有。

已被 Google Vertex AI、Hugging Face、LangChain 等采纳推进。

ANP：去中心化身份 + 语义网

每个智能体有基于 W3C DID 的"数字护照"，可在全球范围自我发现和对等通信。采用 Linked Data（JSON-LD）、schema.org 等语义网技术，让不同 AI 用共同语义描述自己。身份完全自主，不需要注册中心或平台账号。

目前主要停留在传输和发现层，深度协作和环境数据对接还有待完善。

协议关系：互补，不是竞争

                  ┌─────────────────────────────────┐
                  │         完整 Agent 架构          │
                  │                                 │
工具层 ← MCP ────│  Agent ──A2A──► 跨平台 Agent    │
                  │    │              │              │
                  │    │              └──ANP──► 全球 Agent 网络
                  └─────────────────────────────────┘
                   （ACP 概念已收敛至 A2A，不再独立发展）
  • MCP 管 Agent 和工具的连接（数据获取层）
  • A2A（含 ACP 概念）管 Agent 间协作（Agent 通信层）
  • ANP 管去中心化 Agent 网络（开放网络层）

面试要点

  1. 不要把四个协议当成竞争关系——它们各管一层，理想状态下互补融合。面试官问"哪个会赢"，回答"它们不是同一个赛道，就像 TCP 和 HTTP 不是竞争关系"
  2. ACP 的现状要主动提——"ACP 仓库已于 2025 年 8 月正式归档，官方声明合并至 A2A。ACP 与 A2A 概念高度重叠，业界以 A2A 为主流标准"，展示你跟得上动态。另外要提一句 OpenClaw 的 ACP（Agent Client Protocol）是另一个同名协议，避免混淆
  3. ANP 是面试加分项——大多数人只知道 MCP 和 A2A，提到 ANP 和去中心化身份（W3C DID）说明你关注行业前沿
  4. 选型逻辑：工具接入 → MCP；Agent 间协作 → A2A；去中心化场景 → ANP。ACP 概念已收敛至 A2A，新项目直接用 A2A。企业内部最常用的是 MCP + A2A 组合
  5. 行业趋势：协议正在走向兼容和融合，类似当年互联网协议的统一过程。越来越多开源中间件在屏蔽协议差异，为开发者提供统一 API

选型决策树

面试常问"你们为什么选 X 不选 Y"，核心是看你的决策过程而非结论：

你的 Agent 需要什么？
├── 接入外部系统/工具 → MCP（目前唯一成熟选项）
├── 和其他 Agent 协作
│   ├── 同框架内 → 框架内部机制（ADK Task Agent / LangGraph sub-graph / OpenClaw Session Spawn）
│   ├── 跨框架、跨团队、同企业 → A2A
│   └── 跨企业、去中心化 → ANP（概念阶段，暂无生产案例）
└── 既有 MCP 又有 A2A → 典型企业架构（不冲突，互补）

面试话术："我们选 MCP + A2A 组合，因为它们解决不同层面的问题——MCP 管 Agent 和工具的连接，A2A 管 Agent 之间的协作。就像微服务架构里，服务内部用本地调用，服务之间用 gRPC，不是二选一。"

Q6: ADK 中的 Task Agent 是什么？多 Agent 系统如何通信？

回答

Task Agent 是什么

ADK 的 Task Agent 是一种特殊的 Agent，核心能力是委托任务给其他 Agent。它不是"更聪明的 Agent"，而是"任务调度器"——自己不干活，根据用户意图动态决定把活派给谁。

注意：以下代码为概念示意。ADK 的实际 API 以 Agent 为核心类，Task Agent 通过 Agent 的 sub_agents 属性和 Task API 实现（"task agents as workflow nodes"），具体类名和构造方式请以 ADK 官方文档 为准。

from google.adk import Agent

# 普通 Agent：自己干活
researcher = Agent(
    name="researcher",
    instruction="你是研究员，负责搜索和分析数据。",
    tools=[MCPToolset(connection_params=research_mcp_params)],
)

writer = Agent(
    name="writer",
    instruction="你是写手，负责根据研究结果撰写报告。",
    tools=[MCPToolset(connection_params=doc_mcp_params)],
)

# Task Agent：派活，自己不干活
coordinator = Agent(
    name="coordinator",
    sub_agents=[researcher, writer],
)

两种委托模式

1. 多轮任务模式（Multi-turn Task）

子 Agent 可以多轮交互后再返回结果，适合需要反复沟通的复杂任务。

用户 → coordinator → 委托给 researcher
                        │
                        └── researcher 多轮执行（搜索→分析→整理）
                              │
                              └── 返回结果给 coordinator
                                       │
                                       └── coordinator 把结果交给 writer
                                              │
                                              └── writer 生成报告 → 返回最终结果

2. 单轮受控输出（Single-turn Controlled Output）

子 Agent 一次机会必须返回结果，适合简单明确的子任务。

Task Agent vs Workflow

ADK 2.0 有两种编排方式：

维度 Task Agent Workflow
决策者 LLM 动态决定派给谁 开发者预定义图结构
路由 LLM 根据意图判断 条件边硬编码
灵活性 高，可动态调整 低，流程固定
可预测性 低，依赖 LLM 判断 高，每次走同样路
适用场景 任务类型多样，不好预判 流程固定，每步确定 
# Task Agent：LLM 决定路由
coordinator = Agent(
    name="coordinator",
    sub_agents=[researcher, writer, translator],
    instruction="研究问题交给 researcher，写报告交给 writer，翻译交给 translator。",
)

# Workflow：开发者决定路由
root_agent = Workflow(
    name="pipeline",
    edges=[
        ("START", research_agent, writer_agent),  # 固定：先研究再写
    ],
)

混合模式（生产中最常见）：纯用 Task Agent 或纯用 Workflow 都有短板——Task Agent 不可预测，Workflow 不够灵活。实际生产中常见的是外层 Workflow 做流程编排，内层 Task Agent 做灵活路由。例如：Workflow 定义"先调研再写报告"的大流程，但调研阶段内部用 Task Agent 动态决定调哪个研究员。这样既保证流程骨架可控，又保留节点内的灵活性。

Agent 间的通信机制

ADK 中 Agent 间通信的本质是消息传递，不是共享内存，不是远程调用。三种方式：

方式1：Task Agent 的任务委托（最常用）

Task Agent 通过 task message 把意图和上下文传给子 Agent。子 Agent 看不到主 Agent 的完整上下文，只看到 task message。

# coordinator 自动生成 task message 传给子 Agent
# 子 Agent 收到：
{
    "role": "user",
    "parts": [{"type": "text", "text": "请搜索2026年Q1中国新能源汽车销量数据"}]
}
# 子 Agent 返回：
{
    "role": "agent/researcher",
    "parts": [{"type": "text", "text": "根据xxx数据，2026年Q1..."}]
}

方式2：Workflow 的状态传递

前一个 Agent 的输出自动成为后一个 Agent 的输入。Workflow 维护全局状态，每个节点可读写：

def research_callback(ctx):
    result = do_research(ctx.user_input)
    ctx.state["research_result"] = result  # 写入状态，下一个 Agent 可以读

方式3：共享 Session（不推荐）

多个 Agent 共享对话历史，但 ADK 不鼓励——Agent 之间应通过消息通信而非共享状态，共享 session 会导致耦合、破坏独立性。

完整示例

from google.adk import Agent
from google.adk.tools.mcp_toolset import MCPToolset, StreamableHTTPConnectionParams

researcher = Agent(
    name="researcher",
    model="gemini-2.5-flash",
    instruction="你是研究员。根据用户问题搜索和分析数据，返回结构化的研究结果。",
    tools=[MCPToolset(connection_params=StreamableHTTPConnectionParams(
        url="http://knowledge-service.internal:9001/mcp/kb/mcp",
    ))],
)

writer = Agent(
    name="writer",
    model="gemini-2.5-flash",
    instruction="你是写手。根据研究员提供的数据，撰写专业报告。",
    tools=[MCPToolset(connection_params=StreamableHTTPConnectionParams(
        url="http://doc-service.internal:9001/mcp/doc/mcp",
    ))],
)

coordinator = Agent(
    name="coordinator",
    sub_agents=[researcher, writer],
    instruction="""你是任务协调者。根据用户需求分配任务：
1. 研究/分析 → 委托给 researcher
2. 写报告 → 先让 researcher 收集数据，再让 writer 撰写
3. 简单问题 → 自己直接回答，不委托
注意：让 writer 写报告时，要把 researcher 的结果作为上下文传给 writer。""",
)

面试要点

  1. Task Agent 的本质是 LLM 驱动的路由——不是硬编码 if-else，而是 LLM 根据意图动态决定派给谁。这是和 Workflow 的核心区别。注意 ADK 实际 API 中 Task Agent 通过 Agent 的 sub_agents + Task API 实现，没有独立的 TaskAgent 类
  2. Agent 间通信是消息传递，不是共享内存——每个子 Agent 只有 task message 的上下文，看不到主 Agent 的完整对话。保证独立性和可测试性
  3. Task Agent 和 A2A 的区别：Task Agent 是 ADK 框架内部的编排机制（同进程），A2A 是跨框架/跨平台的互操作协议（跨网络）。Task Agent 的子 Agent 必须是同一个 ADK 应用内定义的，A2A 可以连接完全独立的 Agent 服务
  4. 选型：流程固定用 Workflow，流程灵活用 Task Agent，跨服务用 A2A

Q7: 设计一个生产级 Agent 系统的整体架构

这道题是前面所有问题的综合——面试官想看你是否能把 MCP、A2A、Task Agent、鉴权、可观测这些点串成一张完整的图。

典型架构

用户（飞书/企微/Web）
      │
      ▼
  API Gateway（鉴权、限流、路由）
      │
      ▼
  Agent Orchestrator
  ├── Workflow（流程骨架：先调研 → 再分析 → 最后输出）
  │     │
  │     └── Task Agent（节点内灵活路由：调研交给谁、分析交给谁）
  │           │
  │           ├── MCP ──→ OA 系统（查审批、提请假）
  │           ├── MCP ──→ 数据平台（跑 SQL、生成报表）
  │           ├── MCP ──→ 飞书（发通知、发卡片）
  │           └── A2A  ──→ 其他团队的 Agent（跨部门协作）
      │
      ▼
  响应用户

横切关注点

架构图画完了还不够，面试官一定会追问"生产环境怎么做"。横切四个层面：

层面 要点 具体做法
鉴权层 MCP Server 防未授权调用 MCP OAuth 2.1 + API Gateway 层 token 校验；A2A 侧 OAuth 2.0
可观测层 工具调用全链路追踪 MCP Server 端 audit_tool 装饰器记录入参/出参/耗时/状态；LLM 推理链上报 LangSmith / LangFuse；关键指标：工具调用次数、P95 延迟、工具选择准确率
限流层 防止 Agent 疯狂调用工具刷爆下游 MCP Server 端 rate limiting（按调用方身份 + 工具名维度）；API Gateway 层全局限流
降级层 MCP Server 挂了怎么办 Agent instruction 中写明兜底话术（如"数据服务暂时不可用，请稍后再试"），避免无限重试；多 MCP Server 实例 + 健康检查

面试要点

  1. 先画架构图，再讲横切关注点——架构图展示你对全局的理解，横切关注点展示你的生产经验。只画图不讲横切 = 做过 Demo，两者都有 = 做过生产
  2. Orchestrator 是核心——它决定了 Agent 系统是"一锅粥"还是"有条理"。Workflow 提供流程骨架的确定性，Task Agent 提供节点内的灵活性，两者结合是生产中最常见的做法
  3. MCP 和 A2A 不冲突——MCP 管 Agent↔︎工具，A2A 管 Agent↔︎Agent，它们是同一个架构中不同层面的组件。面试中主动点出这一点，比等面试官追问更好
  4. 可观测是第一优先级——Agent 系统最大的挑战是"出了问题不知道哪一步错了"。没有可观测，其他都是空谈。面试中如果只提一个横切关注点，提可观测
